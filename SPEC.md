# 배송마스터 설치형 개발 SPEC 초안

_작성일: 2026-05-17 / 담당: 코과장_

## 1. 목표

배송마스터를 현장용 임시 웹앱이 아니라, 작업자가 휴대폰에 설치해 장기간 안정적으로 쓰는 로컬 우선 설치형 배포판으로 재개발한다.

기존 시즌1/시즌2 앱은 데이터 구조와 업무 흐름을 참고하되 그대로 확장하지 않는다. 새 설치형 버전은 하루 업무 타임라인을 단일 원본 데이터로 삼고, 로그/리포트/통계/조회 화면은 이 원본에서 파생한다.

## 2. 제품 원칙

- 데이터 망실 방지를 최우선으로 둔다.
- 하루 업무 기록은 타임라인 이벤트 중심으로 저장한다.
- 화면 표시용 로그, 리포트 텍스트, 통계 숫자는 원본 데이터가 아니라 파생 결과로 취급한다.
- 오프라인 상태에서도 업무 시작부터 종료, 리포트 생성, 백업까지 가능해야 한다.
- 앱 업데이트, 캐시 갱신, 설치형 배포 재설치 과정에서 기존 데이터를 보존해야 한다.
- 시즌1/시즌2 데이터 마이그레이션은 원본을 덮어쓰지 않고 복사본 또는 새 저장소에만 수행한다.
- 레거시 앱의 구조를 따라가지 않는다. 레거시는 업무 흐름과 데이터 복구 단서만 제공한다.
- 모듈 간 직접 참조를 줄이고, 원본 데이터와 파생 계산의 책임 경계를 명확히 둔다.

## 3. 레거시 참조 요약

### 시즌1

- 역할: 원본/복구 기준, 기존 데이터 구조 확인, 시즌2와 비교.
- 주의: 직접 수정 금지. 내보내기 형식 확인은 복사본으로만 진행.

### 시즌2

- 역할: 최근 현장 테스트 흐름과 긴급 패치 참고.
- 확인된 특징:
  - localStorage 접두어: `dm2_`
  - 에러 로그 키: `dm2_error_logs`
  - 로그, 리포트, 상태 구조가 일부 섞여 있음
- 설치형에서는 이 혼합 구조를 분리한다.
- 레거시의 엉성한 모듈 결합과 임시 패치 누적 구조는 재사용하지 않는다.
- 시즌1/시즌2 코드는 기능 요구와 마이그레이션 단서를 확인하는 읽기 자료로만 사용한다.

## 4. 대상 사용자와 사용 환경

- 주 사용자: 현장에서 배송 업무를 직접 수행하는 작업자.
- 주 기기: Android 모바일 설치형 환경. iPhone Safari는 후순위 호환 대상으로 둔다.
- 주 상황: 이동 중, 신호 불안정, 화면을 오래 보지 못하는 환경.
- 핵심 요구: 빠른 입력, 실수 복구, 날짜별 조회, 안정적인 백업.

## 5. 기본 업무 흐름

1. 진접 출발
2. 청량리 도착
3. 구역 시작
4. 정리 시작/정리 완료 또는 바로 배송 시작
5. 구역 완료
6. 이벤트/도우미 기록
7. 업무 종료
8. 리포트 생성
9. 백업/내보내기

각 단계는 버튼 입력으로 이벤트를 남기며, 필요하면 수동 시간 수정과 메모를 허용한다. 수정 이력은 원본 이벤트를 덮어쓰기보다 보정 이벤트 또는 변경 메타데이터로 남긴다.

## 6. 핵심 데이터 모델

### 6.1 DayRecord

하루 업무의 최상위 원본 단위.

- `schemaVersion`: 데이터 구조 버전
- `date`: `YYYY-MM-DD`
- `status`: `draft`, `active`, `closed`, `reviewNeeded`
- `timeline`: `TimelineEvent[]`
- `zones`: `ZoneRecord[]`
- `helpers`: 도우미 기록
- `adjustments`: 수동 보정 기록
- `meta`: 생성/수정 시각, 기기 ID, 마이그레이션 출처, 복구 상태

### 6.2 TimelineEvent

모든 화면의 기준이 되는 원본 이벤트.

- `id`: 이벤트 고유 ID
- `type`: `depart_jinjeop`, `arrive_cheongnyangni`, `zone_start`, `sorting_start`, `sorting_end`, `delivery_start`, `zone_end`, `helper_add`, `incident`, `day_close`, `manual_adjust`
- `at`: ISO timestamp
- `zoneId`: 구역 이벤트일 때 연결되는 구역 ID
- `payload`: 수량, 도우미명, 이벤트 상세 등 타입별 데이터
- `note`: 사용자 메모
- `createdAt`, `updatedAt`: 입력/수정 시각

### 6.3 ZoneRecord

구역별 수량과 시간 계산의 기준.

- `id`: 구역 고유 ID
- `name`: 구역명
- `order`: 표시 순서
- `counts.total`: 총 수량
- `counts.delivered`: 완료 수량
- `counts.failed`: 미완료/실패 수량
- `counts.extra`: 추가/예외 수량
- `startEventId`, `sortingStartEventId`, `sortingEndEventId`, `deliveryStartEventId`, `endEventId`: 타임라인 이벤트 연결
- `memo`: 구역 메모

### 6.4 파생 데이터

아래 데이터는 저장하더라도 캐시로 취급하며, 원본 변경 시 재생성 가능해야 한다.

- 화면용 로그 목록
- 일일 리포트 텍스트
- 구역별 소요 시간
- 실효율/전체 효율
- 월간 통계
- 검색/필터용 인덱스

### 6.5 원본/파생 데이터 경계

- `timeline`이 하루 업무의 진짜 원본이다.
- `DayRecord.timeline`을 업무 사실의 유일한 원본으로 삼는다.
- `TimelineEvent`는 사람이 실제로 누른 버튼, 입력한 시간, 수량 변경, 도우미 추가, 특이사항을 시간순으로 기록한다.
- 출발, 도착, 구역 시작, 정리 시작/완료, 배송 시작, 구역 완료, 이벤트, 도우미, 업무 종료는 모두 `TimelineEvent`로 기록한다.
- `zones`는 구역 설정과 타임라인 이벤트 연결 ID를 담는 보조 구조다. 구역별 결과 원본이 아니다.
- `ZoneRecord`는 구역명, 순서, 연결 이벤트 ID를 중심으로 유지한다.
- `helpers`는 도우미 기본 정보와 연결 정보를 담는 보조 구조다. 도우미 투입 이벤트의 원본은 `timeline`에 남긴다.
- `report` 또는 리포트 텍스트는 원본이 아니라 `timeline`에서 생성한 출력물이다.
- 화면용 로그와 통계 숫자도 원본이 아니라 출력물 또는 파생 결과다.
- 계산 결과는 원본이 아니라 파생 캐시다. 저장하더라도 `timeline` 변경 시 폐기하고 다시 계산할 수 있어야 한다.
- 수량 원본은 `TimelineEvent.payload`에 남긴다. `ZoneRecord.counts`는 원본으로 쓰지 않는다.
- `ZoneRecord.counts`가 필요하면 화면 성능을 위한 캐시로만 사용하고, `sourceEventIds` 또는 계산 시각을 함께 남긴다.
- 원본과 캐시가 충돌하면 항상 `timeline`을 우선한다.

## 7. 저장소 정책

### 7.1 1차 저장소

- 초기 버전은 IndexedDB를 우선 검토한다.
- 구현 단순성이 더 중요하면 localStorage 기반 어댑터로 시작하되, 저장소 인터페이스는 반드시 분리한다.
- 저장 단위는 날짜별 `DayRecord`로 한다.

### 7.2 백업

- 전체 백업: 모든 날짜의 `DayRecord`, 설정, 앱 버전 정보를 포함한 JSON.
- 날짜별 백업: 단일 날짜 `DayRecord` JSON.
- 자동 백업: 가져오기/마이그레이션/스키마 업그레이드 전 생성.
- 백업 파일에는 `schemaVersion`, `exportedAt`, `appVersion`을 포함한다.

### 7.3 데이터 보존

- 앱 업데이트 시 저장소를 삭제하지 않는다.
- 스키마 변경은 마이그레이션 함수로 처리한다.
- 실패한 마이그레이션은 원본을 유지하고 `needsReview` 또는 `failed` 상태로 남긴다.

## 8. 주요 화면

### 8.1 업무 화면

- 오늘 날짜의 업무 상태 표시
- 단계별 주요 버튼
- 현재 구역, 경과 시간, 수량 요약
- 실수 입력 취소 또는 수정
- 오프라인/저장 상태 표시

### 8.2 로그 화면

- `timeline`에서 생성한 시간순 로그 표시
- 로그 화면은 별도 `logs[]`, `logEntries`, `screenLogs` 같은 저장 데이터를 작성하거나 저장하지 않는다.
- 로그 화면은 `DayRecord.timeline`을 정렬하고 표시 규칙을 적용해 렌더링하는 projection/view다.
- 로그 필터, 검색, 표시용 그룹핑은 UI 상태 또는 파생 결과로만 다루며 업무 원본으로 저장하지 않는다.
- 수동 수정, 이벤트, 도우미, 특이사항 표시
- 날짜별 필터
- 원본 이벤트 상세 확인

### 8.3 리포트 화면

- 하루 업무 종료 후 리포트 텍스트 생성
- 구역별 시간/수량/효율 요약
- 복사 버튼
- 리포트 생성 기준 데이터 확인

### 8.4 날짜 조회 화면

- 날짜별 `DayRecord` 목록
- 상태별 필터: 진행 중, 완료, 확인 필요
- 특정 날짜 복원/수정 진입

### 8.5 월간 통계 화면

- 일자별 총 수량, 총 시간, 평균 효율
- 구역별 누적 통계
- 확인 필요 데이터 제외/포함 토글

### 8.6 설정 화면

- 기본 구역명/순서
- 백업 주기
- 표시 형식
- 데이터 내보내기/가져오기
- 앱 버전/저장소 상태

### 8.7 백업/복구 화면

- 전체 백업 다운로드
- 날짜별 백업 다운로드
- 백업 파일 가져오기
- 가져오기 전 미리보기
- 충돌 발생 시 덮어쓰기/건너뛰기/새 복사본 생성 선택

## 9. 계산 규칙

- 구역 소요 시간은 `zone_start`부터 `zone_end`까지를 기본값으로 한다.
- 정리 시간은 `sorting_start`부터 `sorting_end`까지로 계산한다.
- 배송 시간은 `delivery_start`가 있으면 해당 시점부터 `zone_end`까지로 계산한다.
- `delivery_start`가 없으면 정리 완료 또는 구역 시작 기준으로 보정하되, 보정 여부를 표시한다.
- 실효율은 실제 배송 수행 시간 기준 수량/시간으로 계산한다.
- 전체 효율은 출발/도착/대기/정리 시간을 포함한 기준으로 별도 계산한다.
- 사람이 수정한 시간은 원본 이벤트와 보정값을 구분한다.

## 10. 마이그레이션 요구사항

기존 데이터 후보는 `DATA_MIGRATION.md`를 기준으로 한다.

- 우선순위:
  1. `state.results`
  2. `logs_날짜`
  3. `report.state.logs`
  4. `item.logs`
  5. `logsByDate`
  6. `storageItems` 안의 로그
  7. `reportText`에서 추출 가능한 정보
- 복구 상태:
  - `complete`: 원본 이벤트와 결과가 충분함
  - `partial`: 일부 로그를 재생성함
  - `textOnly`: 리포트 텍스트만 있음
  - `needsReview`: 사람이 확인해야 함
  - `failed`: 복구 불가
- 마이그레이션 UI는 날짜별 결과와 확인 필요 항목을 보여줘야 한다.

## 11. 설치형 요구사항

- 홈 화면 설치 가능
- 오프라인 실행 가능
- 앱 셸 캐싱
- 저장소 데이터는 캐시 삭제와 별개로 보존
- 새 버전 감지 시 사용자에게 업데이트 적용 여부 표시
- 업데이트 전 자동 백업 또는 백업 권장 표시

## 12. 모듈 구조 초안

- `dayStore`: 날짜별 원본 데이터 저장/조회/백업
- `eventTimeline`: 이벤트 추가, 수정, 정렬, 파생 로그 생성
- `deliveryCalc`: 시간, 수량, 효율 계산
- `reportBuilder`: 일일 리포트 텍스트 생성
- `backupImportExport`: 백업, 복구, 충돌 처리
- `legacyMigration`: 시즌1/시즌2 데이터 변환
- `phoneInstall`: 설치, 캐시, 업데이트 상태 관리
- `uiScreens`: 업무, 로그, 리포트, 통계, 설정 화면

### 12.1 모듈 의존 방향

- `uiScreens`는 `dayStore`에서 원본을 읽고, `eventTimeline`, `deliveryCalc`, `reportBuilder`의 결과를 표시한다.
- `reportBuilder`는 `DayRecord`와 `deliveryCalc` 결과만 입력으로 받는다.
- `deliveryCalc`는 저장소나 화면을 직접 알지 않는다.
- `eventTimeline`은 원본 이벤트 생성/수정 규칙만 담당하고, 리포트 문구나 화면 상태를 알지 않는다.
- `dayStore`는 저장과 스키마 마이그레이션만 담당하고, 화면용 로그나 통계를 직접 만들지 않는다.
- `legacyMigration`은 레거시 입력을 새 `DayRecord`로 변환한 뒤 저장소에 넘긴다. 화면/리포트 모듈에 직접 연결하지 않는다.
- `backupImportExport`는 저장소 어댑터를 통해서만 데이터를 읽고 쓴다.

### 12.2 모듈 분리 금지 규칙

- 화면 표시용 로그를 원본 데이터처럼 저장하지 않는다.
- 리포트 텍스트를 원본 데이터로 취급하지 않는다.
- 계산 결과를 여러 저장 위치에 중복 저장하지 않는다.
- 임시 패치를 본 구조에 섞지 않는다.
- UI 상태를 계산 모듈이나 저장소 모듈에 넣지 않는다.
- 마이그레이션 전용 보정 로직을 일반 업무 입력 흐름에 섞지 않는다.

### 12.3 중복 저장 방지

- 같은 사실은 한 곳에만 원본으로 저장한다.
- 업무 이벤트, 시간, 수량 변경의 원본은 `TimelineEvent`다.
- `ZoneRecord`에는 구역명, 순서, 설정값, 연결 이벤트 ID만 저장한다.
- `ZoneRecord.counts`처럼 요약 숫자가 필요하면 파생 캐시로 표시하고 원본처럼 수정하지 않는다.
- `reportBuilder`는 리포트 텍스트를 저장하기보다 매번 `DayRecord`에서 재생성하는 방식을 우선한다.
- 리포트 텍스트를 저장해야 하는 경우에도 스냅샷/내보내기 결과로만 저장하고, 업무 원본으로 역사용하지 않는다.
- `deliveryCalc` 결과는 화면 표시 또는 리포트 생성 입력으로만 사용한다. 계산 결과를 다시 `timeline`에 쓰지 않는다.
- 가져오기/마이그레이션 시 같은 값이 여러 후보에 있으면 원본 우선순위를 기록하고, 불일치 항목은 `needsReview`로 남긴다.

## 14. 비범위

- 서버 동기화는 초기 범위에서 제외한다.
- 여러 사용자 실시간 공유는 초기 범위에서 제외한다.
- 레거시 시즌1/시즌2 코드 직접 수정은 제외한다.
- 리포트 텍스트를 원본 데이터로 삼는 구조는 금지한다.

## 15. 미결정 사항

- 초기 저장소를 IndexedDB로 바로 갈지, localStorage 어댑터로 먼저 갈지 결정 필요.
- 실제 구역명/구역 순서 기본값 확인 필요.
- 효율 계산 공식의 현장 기준 확인 필요.
- 기존 시즌1/시즌2 원본 데이터 샘플 위치 확인 필요.
- 설치형 앱의 배포 방식 결정 필요: 정적 파일, Pinokio/로컬 런처, 별도 호스팅 중 선택.

## 16. 현장 성공 기준

- 하루 업무를 처음부터 끝까지 앱 안에서 완료할 수 있다.
- 하루 업무를 오프라인 상태에서도 끝까지 기록할 수 있다.
- 업무 종료 후 생성된 로그와 리포트가 같은 `timeline` 기준으로 일치한다.
- 수량, 시간, 효율 계산 결과가 리포트와 통계 화면에서 서로 다르게 나오지 않는다.
- 힐스/대체배송 정리 시간이 로그와 리포트에 동일하게 반영된다.
- 오프라인 상태에서도 업무 입력, 저장, 조회, 리포트 생성이 가능하다.
- 앱 업데이트 후에도 기존 날짜별 데이터가 보존된다.
- 전체 백업을 내보낸 뒤 새 환경에서 가져와 같은 날짜 기록을 복구할 수 있다.
- 내보내기 파일만으로 날짜별 원본 데이터를 복구할 수 있다.
- 날짜별 백업을 가져올 때 기존 데이터 덮어쓰기 여부를 사용자가 확인할 수 있다.
- 마이그레이션된 시즌1/시즌2 데이터는 날짜별 복구 상태를 표시한다.
- 마이그레이션 결과는 날짜별로 `complete`, `partial`, `textOnly`, `needsReview`, `failed` 상태를 표시한다.
- 원본과 파생 데이터가 충돌할 때 앱은 `timeline`을 기준으로 복구하거나 확인 필요 상태를 표시한다.
- 현장에서 실수 입력을 수정해도 리포트/로그/통계가 다시 일관되게 계산된다.
