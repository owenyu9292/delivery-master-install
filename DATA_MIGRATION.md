# 배송마스터 데이터 마이그레이션

_SPEC 기준 정렬: 2026-05-17 / 담당: 코과장_

## 1. 목적

시즌1/시즌2에 흩어진 기존 데이터를 설치형 앱의 새 원본 구조인 `DayRecord.timeline`으로 안전하게 옮긴다.

마이그레이션의 목표는 레거시 구조를 보존하는 것이 아니라, 복구 가능한 사실을 새 원본 이벤트로 변환하는 것이다. 로그 화면, 리포트 텍스트, 통계 숫자는 원본이 아니라 복구 단서 또는 검증 자료로만 사용한다.

## 2. 절대 원칙

- 원본 데이터는 직접 수정하지 않는다.
- 가져오기 전 자동 백업을 만든다.
- 마이그레이션 결과는 새 파일 또는 새 저장소에 생성한다.
- `timeline`을 새 앱의 유일한 업무 원본으로 삼는다.
- `zones`, `helpers`, 리포트, 통계는 원본이 아니라 연결/요약/파생으로 취급한다.
- 불확실한 값은 추측하지 않고 `needsReview`로 표시한다.
- 같은 사실이 여러 레거시 후보에 있으면 우선순위와 선택 근거를 기록한다.
- 충돌한 값은 덮어쓰지 않고 날짜별 검토 항목으로 남긴다.

## 3. 변환 대상 새 구조

### 3.1 원본

- `DayRecord.timeline`: 출발, 도착, 구역 시작/완료, 정리 시작/완료, 배송 시작, 도우미, 특이사항, 수량 변경, 업무 종료 이벤트

### 3.2 연결/보조

- `DayRecord.zones`: 구역명, 순서, 설정값, 연결 이벤트 ID
- `DayRecord.helpers`: 도우미 기본 정보, 연결 이벤트 ID
- `DayRecord.adjustments`: 수동 보정과 검토 메모

### 3.3 파생/검증

- 화면용 로그
- 리포트 텍스트
- 효율/통계 계산 결과
- 월간 요약

파생 데이터는 새 원본으로 저장하지 않는다. 필요하면 마이그레이션 검증용 스냅샷으로만 보관한다.

## 4. 기존 데이터 후보

- `state.results`: 구역별 시작/종료/수량/결과 후보
- `state.logs`: 원본 이벤트 또는 이벤트 복구 단서 후보
- `state.events`: 특이사항/이벤트 후보
- `state.helpers`: 도우미 후보
- `logs_날짜`: 날짜별 로그 후보
- `report.state.logs`: 리포트 내부 로그 후보
- `item.logs`: 항목 내부 로그 후보
- `logsByDate`: 날짜별 로그 묶음 후보
- `storageItems`: 백업 안의 저장소 덤프 후보
- `report_날짜`: 리포트 결과 후보
- `reportText`: 텍스트 추출 후보
- `st_날짜`, `all_dates`, `all_data`: 날짜 인덱스/상태 후보

## 5. 원본 우선순위

### 5.1 이벤트/시간

1. 명시적 로그 이벤트: `state.logs`, `logs_날짜`, `report.state.logs`
2. 구역 결과 구조: `state.results`
3. 항목 내부 로그: `item.logs`
4. 날짜별 로그 묶음: `logsByDate`
5. 저장소 덤프 안의 로그: `storageItems`
6. 리포트 텍스트에서 추출한 시간

### 5.2 구역 수량

1. 수량 변경 또는 구역 완료 시점의 명시적 이벤트
2. `state.results`의 구역별 수량
3. 백업/저장소 덤프 안의 구조화된 수량
4. 리포트 텍스트에서 추출한 수량

수량은 새 구조에서 `TimelineEvent.payload`에 원본으로 기록한다. `ZoneRecord.counts`에는 원본 수량을 저장하지 않는다.

### 5.3 도우미/이벤트

1. `state.events`, `state.helpers`
2. 로그에 남은 도우미/이벤트 항목
3. 리포트 텍스트에서 추출한 도우미/이벤트

도우미 투입도 `helper_add` 또는 관련 `TimelineEvent`로 복구한다.

## 6. 변환 규칙

- 날짜별로 하나의 `DayRecord`를 만든다.
- 각 레거시 로그 항목은 가능한 한 하나의 `TimelineEvent`로 변환한다.
- 구역 시작/종료는 `zone_start`, `zone_end` 이벤트로 변환한다.
- 정리 시작/완료는 `sorting_start`, `sorting_end` 이벤트로 변환한다.
- 바로 배송을 시작한 경우 `delivery_start` 이벤트를 만들고 근거를 남긴다.
- 수량은 `TimelineEvent.payload`에 원본으로 기록한다.
- `ZoneRecord`는 구역명, 순서, 연결 이벤트 ID 중심으로 만든다.
- 효율 계산 결과는 저장하지 않고 새 앱의 `deliveryCalc`로 다시 계산한다.
- 리포트 텍스트만 있는 날짜는 추출 가능한 이벤트를 만들되 `textOnly` 또는 `needsReview`로 표시한다.
- 시간이 없는 이벤트는 날짜와 순서를 유지하고 `needsReview`를 붙인다.

## 7. 충돌 처리

### 7.1 충돌 예시

- 같은 구역의 시작 시간이 로그와 결과 구조에서 다름
- 리포트 수량과 구조화된 수량이 다름
- 도우미 기록이 로그에는 있으나 상태 구조에는 없음
- 날짜 인덱스에는 있는데 실제 날짜 데이터가 없음

### 7.2 처리 방식

- 우선순위가 높은 값을 `timeline` 원본으로 채택한다.
- 버려진 후보값은 `adjustments` 또는 migration note에 기록한다.
- 차이가 큰 항목은 `needsReview`로 표시한다.
- 사람이 확인하기 전에는 충돌 값을 평균내거나 임의 보정하지 않는다.

## 8. 날짜별 복구 상태

- `complete`: 원본 이벤트와 결과가 충분하고 충돌이 없음
- `partial`: 일부 이벤트를 재생성했지만 주요 업무 흐름은 복구됨
- `textOnly`: 리포트 텍스트만 있어 구조화 복구가 제한됨
- `needsReview`: 사람이 확인해야 하는 충돌/누락이 있음
- `failed`: 날짜 식별 또는 핵심 데이터 복구가 불가능함

## 9. 마이그레이션 결과 리포트

가져오기 후 날짜별로 다음 정보를 보여준다.

- 복구 상태
- 생성된 `TimelineEvent` 개수
- 생성된 구역 개수
- 도우미/특이사항 개수
- 충돌 항목 개수
- 원본 후보 우선순위와 선택 근거
- 사람이 확인해야 하는 항목

## 10. 검증 기준

- 마이그레이션 후 리포트는 새 `timeline`에서 다시 생성된다.
- 새 리포트와 기존 리포트의 주요 숫자 차이를 비교한다.
- 차이가 있으면 기존 리포트를 원본으로 덮어쓰지 않고 검토 항목으로 남긴다.
- 날짜별 백업을 내보낸 뒤 다시 가져와 같은 `DayRecord`가 복원되는지 확인한다.
- 오프라인 상태에서도 마이그레이션 결과 조회와 백업 내보내기가 가능해야 한다.

## 11. 금지

- 레거시 저장 키를 새 앱의 핵심 저장 구조로 그대로 사용하지 않는다.
- `reportText`를 새 원본으로 삼지 않는다.
- `ZoneRecord.counts`를 수량 원본으로 삼지 않는다.
- 계산 결과를 `timeline`에 다시 쓰지 않는다.
- 마이그레이션 보정 로직을 일반 업무 입력 모듈에 섞지 않는다.
