## 0. 최상위 설계 원칙: 사용자 편의

- 이 앱은 범용 업무 앱이 아니라 오빠가 현장에서 쓰는 개인 현장앱이다. 모든 UI와 기능 판단은 오빠가 실제 현장에서 편한가를 최우선 기준으로 삼는다.
- 버튼, 입력칸, 화면 순서는 코드 구현 순서가 아니라 현장에서 다음에 해야 하는 행동 순서로 배치한다.
- 가장 자주 누르는 다음 행동은 화면 최상단의 주버튼으로 둔다.
- 예외 행동, 우회 행동, 드물게 쓰는 행동은 아래쪽이나 보조 버튼으로 낮춘다.
- 기능을 추가하거나 수정할 때는 먼저 "이 흐름이 현장에서 손이 가는 순서인가"를 확인한다.
- 현장에서는 잘못 누르기, 잘못 입력하기, 무료/유료 오선택이 당연히 생긴다. 기록 수정 기능은 단발 처리로 끝내지 말고 재수정이 가능해야 한다.
- 사용자가 현장에서 직접 수량을 계산해야 하면 실패한 UI다. 가능한 입력은 총합 기준으로 받고, 차감/나머지/효율 계산은 앱이 담당한다.
- 통계 화면에서 미주/힐스테이트/대체배송 비율카드는 보조 정보가 아니라 개발앱의 핵심 지표다. 주간/월간 탐색 화면을 만들 때 비율카드는 가장 먼저 읽히는 위치와 형태로 둔다.
- 통계 배치는 계산/코딩 순서가 아니라 오빠가 현장에서 보고 싶은 순서가 기준이다. 먼저 물량 비율과 편차를 보고, 그 다음 총량/효율/일별 세부를 본다.
- 같은 컴포넌트를 여러 구역이 공유하면 한 구역만 편하게 만들지 않는다. 힐스테이트, 대체배송, 추가구역처럼 같은 UI를 쓰는 화면은 모두 같은 사용자 편의 기준을 적용한다.
- 검수는 버튼이 작동하는지만 보지 않는다. 사용자가 계산, 추론, 스크롤, 재확인을 덜 하게 되었는지 확인한다.

## 0-1. 최상위 작업 전제: 한국어 문서 UTF-8 우선

- 한국어 문서를 읽거나 검수할 때는 처음부터 UTF-8 기준 도구를 사용한다.
- PowerShell `Get-Content` 기본 출력처럼 한글이 깨져 보일 수 있는 방식으로 문서 내용을 판단하지 않는다.
- 한국어 문서 확인은 우선 `node -e`와 `fs.readFileSync(file, "utf8")` 기준으로 한다. PowerShell을 써야 하면 `-Encoding UTF8`을 명시한다.
- 한글이 깨져 보이는 출력이 나오면 그 출력은 작업 근거로 쓰지 않는다. 즉시 UTF-8 기준으로 다시 확인한 뒤 작업한다.
- 깨진 한글 출력이나 추정 복구한 문장을 문서, 보고, TODO, changelog에 복사하지 않는다.
- 이 규칙은 문서 수정뿐 아니라 코드 수정 전 요구사항, TODO, 진행 기록, 미해결 문제, 변경 기록 확인에도 적용한다.

## 0-2. 최상위 작업 전제: 중요 지시 승격과 문서 위치

- 사용자가 "중요하다", "항상", "매번", "앞으로", "반복 방지", "최상단", "코기가 먼저 제안"처럼 작업 방식 자체를 바꾸는 말을 하면 단순 작업 메모가 아니라 최상위 규칙 후보로 판단한다.
- 최상위 규칙 후보는 `todo.md`, `progress.md`, `unresolved.md`, `changelog.md`에만 남기지 않는다. 반드시 `instruction.md` 반영 여부를 먼저 보고한다.
- `changelog.md`는 실제 반영 이력이고, `progress.md`는 현재 작업 흐름이다. 둘 중 어디에도 최상위 규칙을 대신 맡기지 않는다.
- `todo.md`는 할 일 목록이다. 규칙의 최종 위치가 아니다.
- `unresolved.md`는 아직 해결되지 않은 문제 목록이다. 규칙의 최종 위치가 아니다.
- 하위 문서나 과거 문서에서 반복 작업 기준이 발견되면 "하위 문서에 묻힌 최상위 후보"로 보고하고, 승인 후 `instruction.md`로 승격한다.
- 작업 전에는 새 지시가 기능 요청인지, 버그 수정인지, 최상위 규칙 후보인지 먼저 구분한다.

## 0-3. 최상위 데이터 원칙: 기록 보존과 timeline 원본

- 데이터는 코드보다 중요하다. 사용자 기록을 조용히 덮어쓰기, 삭제, 초기화, 마이그레이션하지 않는다.
- 업무 원본은 `DayRecord.timeline`이다. 리포트, 로그, 요약, 수량, 통계는 원본에서 파생한다.
- 리포트 텍스트, 화면 표시 숫자, 통계, 캐시는 원본이 아니다.
- 가져오기, 복구, 초기화, 완료 기록 수정, 앱 업데이트, 서비스워커 캐시 갱신은 데이터 손실 가능성을 먼저 검토한다.
- 위험 작업 전에는 백업 또는 되돌릴 수 있는 기준점을 먼저 확인한다.
- 원본과 파생값이 충돌하면 `timeline`을 우선하고, 불확실하면 조용히 덮어쓰지 말고 확인 필요 상태로 남긴다.
- 정기휴무는 저장 원본으로 자동 생성하지 않는다. 과거 일요일/월요일에 기록이 없으면 통계/조회 화면에서만 가상으로 `정기휴무` 표시하고, 백업 JSON에는 빈 휴무 기록을 새로 만들지 않는다.

## 0-3-1. 최상위 수량 원칙: 구역 순서와 미주 상세 분리

- 구역 수량 계산은 배송지 이름이 아니라 작업 순서를 기준으로 한다.
- 1구역은 입력 수량을 그대로 해당 구역 수량으로 저장한다.
- 2구역 이후는 사용자가 당일 전체 수량을 입력할 수 있으므로, 이전 완료 수량을 먼저 차감해 해당 구역 수량을 확정한다.
- 이 순서 기준 수량 계산은 미주, 힐스테이트, 대체배송, 추가구역 이름과 무관하게 적용한다.
- 미주라는 구역명에 따라다니는 특수 로직은 `1동/2동/3동/나머지` 상세 분해뿐이다.
- 미주가 2구역 이후로 밀려도 먼저 순서 기준으로 미주 구역 수량을 확정하고, 그 확정 수량 안에서 1/2/3동 합계를 뺀 나머지를 계산한다.
- 힐스테이트, 대체배송, 추가구역에는 미주 상세 분해 로직을 적용하지 않는다.

## 0-4. 최상위 검수 원칙: 실제 화면 검수 없이는 완료 아님

- 현장 흐름 변경은 실제 앱 실행 또는 브라우저 접속 검수가 필수다.
- 문법 검사, 타입 검사, 단위 테스트, 코드 리뷰만으로 현장 흐름 작업을 완료 처리하지 않는다.
- 브라우저 검수는 데스크톱 전체 화면이 아니라 오빠가 실제로 쓰는 갤럭시 폴드 접힌 앞 화면에 최대한 가깝게 진행한다.
- 오빠가 보내는 현장 스크린샷은 갤럭시 폴드 앞 화면 기준의 실제 검수 증거로 취급한다.
- 모바일 검수 기본 뷰포트는 실제 기기에서 확인한 CSS 화면값인 411x762 / DPR 2.63 기준으로 진행한다.
- 폭이 더 좁아졌을 때의 깨짐을 확인해야 하는 작업은 보수 검수 옵션으로 360x762 또는 360x800을 추가로 돌린다.
- 430x932 같은 일반 모바일 폭이나 데스크톱 전체 화면만으로 통과한 결과는 현장 UI 검수로 인정하지 않는다.
- 정상 루트만 검수하지 않는다. 현장 실수 루트를 최소 1개 이상 직접 입력해 확인한다.
- 현장 실수 루트는 한 번 바꾸고 끝나는 테스트로 보지 않는다. `대체배송 -> 도우미`, `도우미 무료 -> 유료`, `도우미 -> 구역 복구`, `복구 후 재수정`처럼 되돌림과 재수정을 함께 검수한다.
- 저장 차단은 복구 경로와 같이 설계한다. 막기만 해서 다음 업무 단계로 못 가게 만들면 현장앱 실패로 본다.
- 말도 안 되는 시간과 수량도 검수한다. 종료가 시작보다 빠른 경우, 배송 시간이 1분인데 수량이 과도한 경우, 하루 총량을 뒤 구역에 넣은 경우를 직접 확인한다.
- 검수는 버튼이 작동하는지만 보지 않는다. 화면 순서, 다음 행동 위치, 입력 낭비, 직접 계산 필요 여부까지 확인한다.
- 로그 검수는 시간순만 보지 않는다. 같은 분 안에 이벤트가 겹치면 현장 업무 흐름 기준으로 `이전 구역 완료 -> 다음 구역 시작` 순서가 유지되는지 확인한다.
- 검수하지 못한 항목은 완료가 아니라 `unresolved.md` 또는 `todo.md`에 남긴다.

## 0-5. 공용 작업 트리 / 훅 선별 적용 원칙

- `C:\Codex55Workspace\_ProjectCommonRules`의 공용 규칙은 자동 적용하지 않는다. 이 프로젝트의 세대/폴더 구조에 맞게 선별 적용한다.
- 배송마스터는 현재 Hermes 서버 운영 프로젝트가 아니라 Codex 핸드폰 연동 기반의 현장 PWA/개발앱 프로젝트로 본다.
- 공용 규칙 중 작업 폴더 확인, source-of-truth 확인, 문서 우선, Git 상태 확인, 위험 작업 승인, 실제 사용자 화면 기준 완료 원칙은 적용한다.
- 공용 규칙 중 서버 repo 동기화, systemd 재시작, MCP 서버 운영, Zulip/Hermes 보고서 완료 조건은 기본 적용하지 않는다.
- 개발앱 원본에는 프로젝트 맞춤 `.githooks/pre-commit` 적용을 검토한다. 그대로 설치하지 말고 문서 변경 시 `changelog.md`, 민감정보, `git diff --check`, PWA/cache 변경 여부를 중심으로 조정한다.
- 현재 repo는 source-of-truth와 배포 repo를 겸하므로 소스 수정용 규칙과 배포 산출물 확인 규칙을 함께 적용한다.
- 현장앱 v1/season2에는 현재 hook을 적용하지 않는다. 보호 대상이므로 수정 요청이 명시될 때 별도 적용안을 보고한다.

## 0-6. 최상위 작업 전제: 승인 후 작업 시작 기록

- 사용자가 지시하면 코기는 먼저 이해한 내용을 짧게 보고한다. 길게 늘어놓지 않고 핵심 의도, 영향 범위, 예상 수정/검수만 정리한다.
- 사용자가 그 보고를 보고 승인하기 전에는 코딩, 문서 수정, 파일 생성, 커밋, 푸시, 배포를 시작하지 않는다.
- 승인 직후에는 `progress.md`에 작업 시작 기록을 남긴다. 이 기록은 감시용이 아니라 맥락 압축, 후속 검수, 다른 작업자가 이어받을 때의 기준점이다.
- 작업 시작 기록에는 사용자 지시, 코기 이해, 승인 상태, 예상 수정 파일, 검수 계획을 간단히 남긴다.
- 이후 작업 범위가 바뀌면 기존 승인을 확대 해석하지 않는다. 멈추고 변경 범위를 다시 보고한 뒤 승인받는다.

## 0-7. 최상위 작업 전제: 현장 장애 긴급 대응

- 사용자가 현장 업무 중 앱 입력이 막혔다고 보고하면 일반 작업 절차보다 장애 해소를 우선한다.
- 현장 장애는 `원인 요약 -> 최소 수정 -> 검사/빌드 -> 즉시 배포 -> 현장 확인 요청` 순서로 처리한다.
- 이때 긴 보고, 과도한 계획 설명, 불필요한 브라우저 창 실행은 피한다.
- 이미 사용자가 긴급 배포를 승인한 맥락이면 별도 승인 반복 없이 핫픽스를 진행한다.
- 핫픽스 채널에서 사용자가 현장 긴급 수정이라고 요청하면 현장에서 막힌 부분을 임시로 틀어막는 최소 수정, 검사, 빌드, 배포까지 전량 승인된 것으로 간주하고 최대한 빨리 처리한다.
- 핫픽스는 정식 설계 완료가 아니다. 막은 내용과 남은 정식 수정 항목을 문서에 남기고, 퇴근 후 일반 작업창에서 정식 수정한다.
- 단, 데이터 삭제, 초기화, 기록 덮어쓰기, 기존 현장앱 v1/season2 수정은 긴급 상황에서도 별도 확인 없이는 하지 않는다.
- 긴급 조치 후에는 반드시 `progress.md`, `todo.md`, `changelog.md`에 사후 기록을 남기고, 같은 유형이 전 구역 공통인지 검수 항목으로 확장한다.

## 0-8. 최상위 작업 전제: 핫픽스 채널은 현장 접속용 초경량 채널

- 핫픽스 채널은 긴 기획/회고/구조 정리용이 아니라, 핸드폰 현장 접속 안정성을 위한 초경량 장애 대응 채널이다.
- 일반 작업창은 긴 개발 맥락, 문서 정리, 구조 변경, 검수 계획, 배포 판단을 담당한다.
- 핫픽스 채널은 `HOTFIX_HANDOFF.md`를 기준으로 현재 버전, 작업 폴더, 금지 폴더, 최신 반영 상태만 읽고 움직인다.
- 핫픽스 채널에서는 긴 설명, 전체 회고, 대규모 설계, 불필요한 질문, 반복 로그 덤프를 피한다.
- 핫픽스 채널의 기본 흐름은 `짧은 원인 확인 -> 최소 수정 -> 검사/빌드 -> 배포 확인 -> 현장 확인 요청`이다.
- 핫픽스 채널에서 현장 긴급 수정 요청이 들어오면 별도 승인 질문으로 멈추지 않는다. 요청 자체를 임시 차단/복구용 최소 수정 전량 승인으로 보고 즉시 처리한다.
- 핫픽스 채널은 정식 리팩터링이나 큰 구조 변경을 하지 않는다. 막은 내용과 정식 수정 필요 사항을 `HOTFIX_HANDOFF.md`에 남긴 뒤 일반 작업창으로 넘긴다.
- 핫픽스 채널에서 새로 확인된 상태나 수정 결과는 `HOTFIX_HANDOFF.md`에 짧게 반영해 다음 접속도 같은 기준에서 시작하게 한다.
- 핫픽스 채널이 처리하기 어려운 큰 구조 변경은 일반 작업창으로 넘기고, 핫픽스 채널에는 필요한 최신 상태만 남긴다.
- 핫픽스 채널용 별도 칸반은 만들지 않는다. 일반 작업창과 핫픽스 채널은 같은 작업 보드(`todo.md`, `progress.md`, `unresolved.md`, `changelog.md`)를 사용한다.
- 같은 작업 보드 안에서 기록자/채널을 구분한다. 예: `기록자: 일반 작업창`, `기록자: 핫픽스 채널`, `채널: hotfix`, `채널: main`.
- 핫픽스 채널에서 임시로 막은 문제는 `HOTFIX_HANDOFF.md`에 짧게 적고, 정식 작업 항목은 같은 작업 보드의 `todo.md` 또는 `unresolved.md`에 남긴다.
- 핫픽스 채널은 작업 보드를 따로 관리하지 않지만, 수정 후에는 같은 작업 보드에 반드시 기록해 일반 작업창과 상태가 갈라지지 않게 한다.
## 1. 절대 규칙

- 코딩, 파일 생성, 문서 수정, 폴더 생성, 커밋, 푸시, 배포, 우회 경로 선택은 먼저 보고하고 승인받은 뒤 진행한다.
- 사용자가 현장앱을 명시하지 않으면 현장앱 저장소를 건드리지 않는다.
- `delivery-master-legacy`는 건드리지 않는다.
- 의도한 폴더나 저장소가 없으면 대체 하위 폴더를 만들지 않는다. 멈추고 보고한다.
- 최상위 원칙 `0`부터 `0-4`를 먼저 적용한다.
- 사용자의 지시는 구현 전에 반드시 추적 항목으로 바꾼다.
- 사용자가 새 지시나 수정사항을 말하면 바로 구현하지 않고, 먼저 문제, 기대 동작, 영향 범위, 문서 반영 위치를 요약 보고한다.
- 사용자가 특정 구역이나 특정 화면의 문제로 말해도, 먼저 앱 전체 공통 구조 문제인지 판단한다.
- 특정 증상이 전 구역에 반복될 수 있으면 개별 패치가 아니라 공통 규칙으로 설계한다.
- 수정 전 보고에는 해당 문제가 특정 구역 한정인지, 전 구역 공통인지 반드시 구분한다.
- 사용자가 코더가 아니므로, git, 테스트, 백업, 배포 분리 같은 기본 개발 안전장치가 빠져 있으면 코기가 먼저 제안한다.
- 반복되는 문제가 보이면 단순 수정이 아니라 작업 방식 자체의 누락을 의심한다.
- 개발앱의 원본은 고정된 과거 버전이 아니라 현장에서 사용 가능하다고 확인된 최신 안정 상태다.
- 개발앱 수정 전에는 직전 안정 상태를 git 기준점으로 남기고, 작업은 새 브랜치에서 진행한다.
- 테스트 실패나 현장 사용 불가 문제가 생기면 직전 안정 기준점으로 되돌릴 수 있어야 한다.

## 2. 문서 인코딩 세부 규칙

- 새로 작성하거나 수정하는 핵심 문서는 UTF-8 기준으로 작성한다.
- 이미 깨진 문서는 임의로 원문을 추정해 덮어쓰지 않는다.
- 깨진 문서를 발견하면 `unresolved.md`에 등록하고, 복구 범위는 따로 보고한 뒤 승인받는다.
- 오빠가 읽는 운영 문서는 한국어로 쓴다.
- 파일명, 코드 식별자, 함수명, 테스트명은 기존 영어 체계를 유지한다.

## 3. 작업 전 절차

- 이 문서를 먼저 읽는다.
- 한국어 문서는 UTF-8 기준 도구로 확인한다. 깨져 보이는 출력으로 요구사항을 판단하지 않는다.
- 새 지시가 최상위 규칙 후보인지 먼저 판단한다.
- 최상위 규칙 후보면 `instruction.md` 반영 위치를 먼저 보고하고 승인받는다.
- `unresolved.md`에서 관련 미해결 문제를 확인한다.
- 코기가 이해한 핵심, 영향 범위, 예상 수정 파일, 검수 계획을 오빠에게 짧게 보고한다.
- 승인 전에는 작업하지 않는다.
- 승인 직후 `progress.md`에 작업 시작 기록을 남긴다.
- `todo.md`에 구체 작업 항목을 추가하거나 갱신한다.

## 4. 작업 중 절차

- 수정한 파일과 중요한 판단을 `progress.md`에 기록한다.
- 새 문제가 발견되면 `unresolved.md`에 추가한다.
- 작업 범위가 바뀌면 멈추고 다시 보고한다.

## 5. 작업 후 절차

- `todo.md`에서 완료/미완료 상태를 갱신한다.
- `unresolved.md`에서 남은 문제를 갱신하거나 해결 근거를 남긴다.
- `changelog.md`에 실제 수정 파일, 테스트 결과, 브라우저 검수 결과, 완료 판정을 기록한다.
- 오빠에게 변경 내용, 검증 내용, 남은 위험, 배포 대기/완료 상태를 보고한다.

## 6. 현장 흐름 검수 기준

현장 흐름을 바꾸는 작업은 아래를 확인해야 한다.

- 정상 루트: 하루 업무가 처음부터 끝까지 정상 진행되는가.
- 실수 루트: 흔한 현장 실수가 기록을 조용히 망가뜨리지 않는가.
- 수정 루트: 완료된 기록을 수정할 때 사용자가 직접 계산하지 않아도 되는가.
- 리포트 루트: 총합, 구역 수량, 비율, 효율이 현장 감각과 맞는가.
- 저장 루트: 새로고침, 재접속, 앱 재시작 후 기록이 살아 있는가.
- 백업 루트: 백업/가져오기 문구와 동작이 헷갈리지 않는가.
- UI 배치: 버튼은 코드 상태 순서가 아니라 현장에서 다음에 누를 순서로 배치한다. 가장 자주 누르는 다음 행동은 최상단 주버튼이고, 예외 행동은 보조 버튼이다.
- 공통 UI: 힐스테이트, 대체배송, 추가구역처럼 같은 컴포넌트를 쓰는 화면은 한 구역만 보지 말고 같은 규칙이 전 구역에 적용되는지 확인한다.

## 7. 실제 앱 검수 규칙

- 현장 흐름을 바꾸는 작업은 실제 앱 실행 또는 브라우저 접속 검수를 해야 한다.
- 문장 검수, 코드 검토, 문법 검사, 단위 테스트만으로 완료 처리하지 않는다.
- 최소 1개 정상 루트와 최소 1개 실수 루트를 직접 입력해 확인한다.
- 검수한 시나리오와 결과는 `changelog.md`에 남긴다.
- 검수하지 못한 항목은 완료가 아니라 `unresolved.md` 또는 `todo.md`에 남긴다.
- 구체적인 현재 문제 목록은 `unresolved.md`에서 관리한다.
- 이번 작업에서 검수할 항목은 `progress.md`에 적는다.

## 8. 문서 역할

- `instruction.md`: 절대 규칙과 작업 절차. 오빠가 확인할 최상위 문서.
- `todo.md`: 해야 할 일과 보류한 일.
- `progress.md`: 현재 작업 범위, 수정 파일, 진행 중 판단.
- `unresolved.md`: 아직 고치지 않은 문제.
- `changelog.md`: 실제 반영 내역, 테스트, 검수, 완료 판정.
- 기존 문서도 참고하지만 위 5개 문서를 대체하지 않는다.

## 9. 완료 기준

작업은 아래 조건을 만족해야 완료다.

- 요청한 동작이 구현됐거나 명시적으로 보류됐다.
- 관련 TODO가 갱신됐다.
- 관련 미해결 문제가 갱신됐다.
- 테스트 결과 또는 테스트하지 못한 이유가 기록됐다.
- UI/현장 흐름 변경이면 브라우저 검수 결과가 기록됐다.
- 오빠에게 남은 위험까지 포함해 보고했다.
